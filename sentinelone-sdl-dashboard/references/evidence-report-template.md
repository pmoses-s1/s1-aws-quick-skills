# Log-Evidence Report Template

Every dashboard delivered by this skill ships with a log-evidence report. The report exists to prove that each panel runs against live data, captures actual sample rows, and either renders data or has a documented reason for being empty. A dashboard delivered without this report is incomplete.

The skill produces three files:

1. `<dashboard>.evidence.json`: per-panel evidence keyed on `tab::title`. Generated by `scripts/validate_dashboard.py`. Stays in the repo.
2. `<dashboard>.evidence.md`: human-readable mirror of the JSON. Generated by `scripts/validate_dashboard.py`. Stays in the repo.
3. `<dashboard>.pdf`: leadership deliverable. Generated by `scripts/render_validation_pdf.py`. Distributed alongside the dashboard URL.

This file is the schema and required-content reference for those three artefacts.

---

## 1. Per-panel evidence JSON

`<dashboard>.evidence.json` is a JSON object with one entry per panel. Keys are `tab_name::panel_title` strings. Values are objects with this exact shape:

```json
{
  "ok": true,
  "elapsed_s": 1.42,
  "row_count": 25,
  "columns": ["dst.ip.address", "hits"],
  "sample_rows": [
    ["192.0.2.10", 1842],
    ["198.51.100.5", 1206],
    ["203.0.113.7", 998]
  ],
  "matchCount": 1234567,
  "omittedEvents": 0,
  "verdict": "PASS"
}
```

Required keys for every entry:

- `ok` (bool): did the query execute?
- `elapsed_s` (number): wall-clock time for the API call.
- `verdict` (string): one of `PASS`, `WARN`, `EMPTY`, `FAIL`, `SKIP`.

Required when `ok=true`:

- `row_count` (int): rows in the response. 0 is a valid datapoint.
- `columns` (string[]): column names in the response, in order.
- `sample_rows` (array): first 3 rows verbatim. This is the log evidence.
- `matchCount` (int|null): events scanned before grouping (`data.matchCount` or `matchingEvents`). Distinguishes "no data" from "filtered out."

Required when `ok=false`:

- `error` (string): first 300 chars of the exception message.

Optional when applicable:

- `omittedEvents` (int): events scanned but dropped due to row-cap.
- `query` (string): the panel's PowerQuery body, copied verbatim. The PDF renderer prints this when present.

## 2. Markdown evidence file

`<dashboard>.evidence.md` mirrors the JSON in human-readable form. Required structure:

```markdown
# Dashboard validation evidence: `<dashboard>.json`

- Source dashboard JSON: `<path>`
- Time window: `start_time=7d`, `end_time=now`
- Generated: `<timestamp>`

## Summary

- Total panels: <N> (runnable: <R>, markdown/skipped: <S>)
- PASS: <p>, WARN: <w>, EMPTY: <e>, FAIL: <f>

## Tab: <name>

### <panel title>  ·  `<style>`  ·  **<verdict>**

- elapsed: `<x>s`
- row_count: `<n>`  ·  matchCount: `<m>`
- columns: `[...]`

```text
<exact PowerQuery body>
```

Sample rows (first 3):

```json
[ ... ]
```

(repeat per panel)

## Appendix: empty-result panels

(list every panel where verdict == EMPTY, with matchCount and the operator's
SOC-meaningful interpretation)
```

The Appendix is mandatory whenever any panel returns 0 rows.

## 3. PDF report

`<dashboard>.pdf` is the leadership deliverable. It must contain:

### Cover page

- Dashboard title.
- Generation timestamp.
- Source filename.
- Summary table with these rows: Total panels, PASS, WARN, EMPTY (0 rows), FAIL, Skipped.
- One paragraph explaining the report's purpose ("This report is the live-data evidence base for the dashboard...") and that EMPTY is valid evidence when documented.

### Per-tab sections

For every tab containing at least one runnable panel, a section page that contains, per panel:

- Title (H2).
- Verdict (coloured), style, elapsed, rows, matchCount on a single metadata line.
- Error message in red if `verdict == FAIL`.
- The PowerQuery body in a fixed-width box (when `query` is present in the evidence record).
- A sample-rows table: header row from `columns`, up to 8 columns, up to 3 data rows, with cell text truncated at 80 chars. Nested cells (dicts/lists) flattened with `json.dumps`.
- For empty panels: an amber callout "NO LOG EVIDENCE in the queried window" with the matchCount.

### Appendix: empty-result panels

A single table with three columns:

| Panel (`tab :: title`) | matchCount | Interpretation |

Interpretations are sourced from a sidecar `interpretations.json` (a flat object keyed on the same `tab::title`). Any empty panel without an interpretation MUST be rendered with a placeholder string so the operator sees what is missing. The acceptable placeholder is exactly: `"TODO: document SOC-meaningful interpretation for this empty panel."`

The Appendix exists because a panel returning 0 rows without explanation is indistinguishable to the reader from a broken panel. Every empty entry is flagged for follow-up so it gets documented before the next iteration of the dashboard ships.

## 4. Optional sidecar: interpretations.json

```json
{
  "Threat Activity::Allowed malicious URLs (warn)": "0 here is the desired posture. Non-zero indicates a regression in the URL-rewriting policy worth investigating within 24h.",
  "Identity::Distinct users with detected events (24h)": "0 here is the desired posture. Non-zero indicates at least one user has a detected authentication anomaly; correlate with EDR alerts and locked-account events.",
  "Phish::Phishing-classified emails delivered (events)": "Non-zero baseline expected (a noisy delivery channel). Spike investigation triggers at >2× rolling 7-day median."
}
```

Pass this file to `render_validation_pdf.py` with `--interpretations`. Any panel key present here will have its placeholder text replaced. Any empty-result panel key NOT present here renders with the TODO placeholder so the operator sees the gap.

---

## 5. Verdict rules (used by both validate_dashboard.py and the PDF)

| Panel style | Pass condition |
|---|---|
| `number` | `row_count == 1` and `len(columns) == 1` (PASS), 0 rows = EMPTY, multi-row = WARN |
| `donut` / `pie` | `row_count >= 1` and `len(columns) >= 2` (PASS), 0 rows = EMPTY, single column = WARN |
| `stacked_bar` / `bar` / `line` / `area` / `honeycomb` | `row_count >= 1` and `len(columns) >= 2` (PASS), 0 rows = EMPTY, single column = WARN |
| `table` (or unspecified) | Any `ok=True` row count = PASS (sample rows captured); 0 rows = EMPTY |
| `markdown` | SKIP (no query, no API call) |

`FAIL` is reserved for `ok=false`. It always exits the validator non-zero so CI pipelines can gate deploy on this.

---

## 6. Worked example: how the report looks for a passing 12-panel dashboard

```
Total panels: 12 (runnable: 10, markdown/skipped: 2)
PASS:  8
WARN:  0
EMPTY: 2          <- panels with row_count == 0
FAIL:  0
```

In this case, the Appendix lists the two EMPTY panels along with the operator's interpretation:

- `Endpoint Health::Endpoints offline >7d`: "0 is desired (full agent reach). Non-zero requires a per-site agent-health follow-up."
- `Phish::Malicious URLs delivered (warn)`: "0 reflects the URL-rewriting policy hard-blocking. Non-zero indicates a regression worth a same-day RCA."

The PDF can then be sent to leadership with confidence that every empty panel is meaningful posture, not a broken dashboard.
