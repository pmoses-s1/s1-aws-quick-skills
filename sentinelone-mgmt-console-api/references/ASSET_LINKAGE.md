# Asset Linkage on Ingested UAM Alerts

Empirical reference for how `assets[]` populates on alerts posted through
the UAM Alert Interface (`ingest.us1.sentinelone.net`), versus alerts
generated by the SentinelOne agent itself. Based on testing on
`your-tenant.sentinelone.net` in April 2026.

## Summary

Every ingested alert gets at least one `assets[]` entry derived from
`resources[]` on the posted OCSF envelope. What differs is how that
asset is classified and whether it links to real tenant inventory.

Key findings:

1. `assets[].agentUuid` is **never** populated for alerts posted via the
   Alert Interface. It is only populated when an alert originates from
   an S1 agent installed on the tenant. No OCSF field combination
   (tested 14 variants including `resources[].agent_list`,
   `device.agent_uuid`, `device.uuid` matching real agent UUIDs,
   top-level `agent`/`endpoint` objects) causes the UAM server to
   reconcile the ingested asset to tenant inventory.

2. `assets[].category` and `assets[].subcategory` are driven by
   `metadata.product` / `metadata.vendor_name` on the alert envelope,
   **not** by `device.type_id` on the indicator or `resources[].type_id`
   on the alert. This is surprising because OCSF has `device.type_id`
   specifically for this purpose; UAM ignores it on the ingest path.

3. `resources[]` with multiple entries spanning different devices
   causes the alert to be **silently dropped** by the stitcher. A single
   alert must target a single resource. If you need per-device alerts,
   post separate alerts.

4. Indicators can freely reference distinct `device.uid` values within
   the same batch and will all stitch to one alert via
   `finding_info.related_events[].uid`, but the alert's own
   `resources[]` still has to be a single entry (use the first
   indicator's device, or a synthetic umbrella host).

## Classification matrix

Tested combinations of `metadata.product.name` and
`metadata.product.vendor_name` on the alert envelope, observing the
resulting `assets[].category` / `subcategory` via UAM GraphQL:

| product name     | vendor_name      | category | subcategory    |
| ---------------- | ---------------- | -------- | -------------- |
| smoke-product    | smoke-vendor     | Device   | Other Device   |
| SentinelOne      | SentinelOne      | Server   | Virtual Machine|

The classifier is string-matched against a known-products table on the
server. Unknown product names fall through to "Device / Other Device".
Known S1 product names get promoted to infrastructure categories. The
vendor field alone is not enough; both fields need to match the
server's product table.

`osType` tracks `device.os.type_id` on the referenced indicator when
that is set (1=Linux, 200=Windows, 300=Mac). If the indicator omits
`device.os`, `osType` comes back as "Unknown OS".

## What populates per call path

| Field                        | Ingested alert                 | Agent-generated alert         |
| ---------------------------- | ------------------------------ | ----------------------------- |
| `assets[].id`                | synthesized from device.uid    | real inventory ID             |
| `assets[].name`              | `device.hostname`              | agent hostname                |
| `assets[].agentUuid`         | **always null**                | real S1 agent UUID            |
| `assets[].category`          | driven by metadata.product     | derived from agent inventory  |
| `assets[].subcategory`       | driven by metadata.product     | derived from agent inventory  |
| `assets[].osType`            | from indicator.device.os.type_id | from agent                  |
| `assets[].accessible`        | false                          | true (if agent is online)     |
| `assets[].primary`           | true                           | true                          |
| `assets[].agentVersion`      | null                           | real agent version            |
| `assets[].lastLoggedInUser`  | null                           | from agent                    |

## Recommended patterns

### Demo / smoke ingest (no real assets on tenant)

Use defaults. `build_alert_referencing` will post with
`product="smoke-product"` and the resulting asset will render as
"Device / Other Device". This is appropriate when you want the alert to
be visibly distinguishable from production agent-generated alerts.

### Demo / presenting as an S1-originated alert

```python
alert = build_alert_referencing(
    alert_uid=...,
    indicators=[...],
    now_ms=now_ms,
    title=...,
    description=...,
    detection_product="SentinelOne",
    detection_vendor="SentinelOne",
)
```

This will render as "Server / Virtual Machine" and look much closer to
a real agent-generated alert in the UAM UI. Still no `agentUuid`
linkage, so the Assets tab will not navigate to a live endpoint record.

### Genuine agent linkage

Not possible via the Alert Interface. If you need the alert to link to
a real endpoint record (`agentUuid` populated, Assets tab navigates,
isolation actions available), the alert must originate from an S1
agent installed on the tenant (real detection, STAR custom rule hit,
or Hyperautomation-driven `sendCustomEvent`). No known OCSF field
shape reconciles against tenant inventory on the ingest path.

## Probing your own tenant

```python
from scripts.s1_client import S1Client
from scripts import unified_alerts as ua

c = S1Client()
# Default fields now include assets
alert = ua.get_alert(c, alert_id="<your alert id>")
print(alert.get("assets"))
```

The default `get_alert` selection as of this skill version includes
`assets { ... }` and `dataSources { id name }`. If you only need core
fields on a hot path, pass `fields=ua._ALERT_CORE_FIELDS`.

To sweep many alerts, `list_alerts` and `paginate_alerts` still default
to `_ALERT_CORE_FIELDS` (cheap). Pass
`fields=ua._ALERT_DETAIL_FIELDS` when you actually need the asset join
on every edge.
